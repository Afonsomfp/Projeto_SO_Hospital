The system is composed of multiple processes and threads that simulate patient triage and their subsequent treatment by doctors according to different priorities and timings, using various IPC mechanisms such as named pipes, semaphores, shared memory, message queues, signals, and memory‑mapped files, ensuring proper synchronization and correct simulator operation. My program consists of a total of eight C source files, with Projeto_SO being responsible for the entire functioning and processing of the software. This is where all IPC mechanisms are created/initialized, including named pipes, shared memory, message queues, signals, memory‑mapped files, and semaphores, as well as the creation of doctor processes and triage threads.

The performance of my project is based on a loop that only ends when Ctrl‑C is pressed. Therefore, while Ctrl‑C is not triggered to terminate the program, it continuously checks whether new doctors need to be created, including the extra doctor, and also monitors the arrival of elements through the pipe using a timeout.

One of the most important components is the triage process. After receiving patients from the named pipe, they are placed in a queue to be triaged. A signal is then sent to the condition variable used by the triage threads. As soon as a thread is awakened by this signal, it removes a patient from the queue and performs the corresponding triage, storing all essential data required for statistical calculations. Since the number of threads can be changed, if the number is reduced to a value lower than the current one, the thread’s ID is used to terminate its execution. In the opposite situation, new threads are created.

Another essential component for the functioning of this project is the work performed by the doctor process. It creates a thread responsible for counting the shift time, which will notify the process by sending a signal (SIGALRM) when the shift ends. While the shift has not ended and the program has not received a termination signal, the process waits for a patient to be sent from triage for treatment. Once a patient arrives, the doctor performs the treatment and stores all necessary data for statistical calculations. When the shift ends (and consequently the process must terminate), a SIGCHLD signal is sent to the admission process, and a new doctor process is created. Additionally, an extra doctor process may be created if the number of waiting patients exceeds 80% of the maximum queue capacity.

All events occurring during the execution of the project are recorded in the text file DEI_Emergency.log, with the corresponding logging functionalities implemented in Log.c. In summary, the developed system provides a hospital emergency simulator that covers the entire workflow from patient admission to treatment, using multiple processes, threads, and IPC mechanisms, ensuring efficient management of data and resources and the correct collection of statistical information about the execution.

Note: To change the number of available threads during program execution, the special command sent through the named pipe to modify the TRIAGE parameter is, for example: echo "TRIAGE 10" > /tmp/input_pipe
